Composition and Orchestration

As explained earlier, ensuring that containers hold a single application or service is good practice. But how to use such limited-purpose containers to set up a complex system comprised of multiple applications, models, and services as is the case for ForestNavigator? This requires a means of composing containers into a coherent whole that arranges for things such as:

  • Which containers instances derived from what images should participate.

  • Shared access to file volumes.

  • One or more virtual networks shared between containers.

  • Ensuring that containers providing services are started before containers using those services.

  • Ensuring that all containers in a composition can be jointly started and stopped.

  • Exposing containers to the Internet, possibly through a load-balancer/proxy that itself can occupy a container.

  • …

Container composition can be thought of as configuring such a shared arrangement. Container orchestration can be thought of as dynamic composition, providing for things such as adding container instances as load increases, recovery from failure, and so on.

Because containers hide the software they contain, the tools used for composition and orchestration can be standard and general-purpose: these tools only need to know how to handle containers from the “outside”, that is, interact with containers through a standard API that is the same for all containers. This is analogous to how shipping containers containing any kind of goods can be handled by a container crane at a container harbor.

Docker Compose

Docker Compose is a standard tool for composing Docker containers. It is either included in a Docker installation, or can be installed as an additional package. Docker Compose is basically an alternative to the Docker command line interface. Instead of command parameters, Docker Compose takes as input a declarative configuration file that specifies how to set up or tear down a collection of containers, their networks, and volumes. Docker Compose is limited to managing containers running on a single machine. As such, it can help with testing a collection of collaborating containers during development and can suffice for simple production deployments that are not too demanding.

Kubernetes

Kubernetes is a widely-used toolkit for orchestrating containers. It is the deployment framework chosen for ForestNavigator. Kubernetes can orchestrate containers on multiple machines so that the system can scale beyond what a single machine can provide. Also, Kubernetes can dynamically adapt to change so that the deployment is robust and responsive. Kubernetes can migrate containers amongst machines so as to make the best use of available resources. It can even be set up such that a machine can fail without causing interruption of service.

Kubernetes is a much more complicated system than Docker Compose. Setting it up is a rather involved process. It is therefore not recommended to invest the time to manually set up a Kubernetes installation on your development machine so as to help with the development and testing of multi-container applications. However, there are tools that provide Kubernetes in ready-to-go form or that automate its installation: some examples are Kind, Minikube, K3s, and also Docker Desktop which includes a standalone Kubernetes server and client. This will allow you to perform testing of deployment to Kubernetes on your local development machine. Also, a managed Kubernetes cluster can be rented from several cloud providers.

Beware though that this is probably overkill for your use case. In case of single-container software, you can build and test with Docker alone. In case of a simple multi-container application, adding Docker Compose will help and be sufficient until the very last deployment step.

Kompose

To bridge the gap between Docker Compose and Kubernetes, the Kompose tool is available. It translates a Docker Compose configuration file to Kubernetes Resources.
