Introduction

Once an application or collection of collaborating applications has been developed and containerized, it is time for deployment. Unlike development, which typically occurs on a local development machine, deployment will usually target one or more remote servers where the containerized software is exposed to its users or its usage context. Containers can be directly deployed into “production”, but it is good practice to first deploy onto a test cluster for integration testing. This allows issues to be caught that only crop up once containers start to interact and access shared resources.

The Kubernetes-based container deployment approach chosen for ForestNavigator abstracts away much of the underlying complexity and hard work involved in server-side deployment. The physical servers, network, and other hardware are not visible: though this physical infrastructure is not managed by Kubernetes and still needs to be administered otherwise, Kubernetes allows it to be ignored from the perspective of container deployment. Instead, attention can be focused on specifying the desired state of a collection of containers. The Kubernetes framework thereafter takes care of orchestrating the containers to attain that desired state and re-attain it in case of change such as a failure or increase in demand.

Fortunately, when developing containerized software for ForestNavigator deployment, it is not necessary to have a full understanding of Kubernetes. It suffices to know how to develop and test containerized software such that it is ready for deployment in a Kubernetes context. In this section, we will therefore focus just on the concepts and aspects that you need to know.
