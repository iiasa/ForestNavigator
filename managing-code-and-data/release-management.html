Release Management

Once your code or data has evolved to a state of readiness, you can release it. How to do so depends on the audience and intended use of the code or data. For example, you may release for:

  • Deploying code into production.

  • Making available a new version to users.

  • Providing a dataset to project partners.

  • Release a citable version of a code base or dataset to the public.

  • …

Typically, releases are done by the owner of the code base or dataset. For workflows with multiple collaborators where ownership is shared, a capable collaborator should take on the role of release manager.

Tagging

Using Git’s tagging feature you can attach a label to a commit (a particular version of the tree of files under management) so that you can easily return that that particular version later. When people or automation scripts watching the repository understand the name convention chosen for the tags, that can be enough for a release. Typically, you will want to adopt some versioning scheme to name the tags.

GitHub/GitLab Release Management

When the repository is hosted on GitHub or GitLab, you can manage releases through the web interface via which the repository is administered. Both GitHub and GitLab allow you to create a tag (see above), write a release description, and have the release show under a permalink together with downloadable assets. This allows people to browse releases and review release notes before downloading release assets. For details see the documentation on GitHub release management and GitLab releases.

Note

No system administrator needs to be involved in this since the contributor that manages the release can connect the release branch in his or her local repository to the GitHub/GitLab repository, prepare the release, and push it out when tested and ready. Then the manager uses the GitHub/GitLab web interface to write the release notes and define the tag.

Zenodo Release Management

Zenodo allows you to publish research outputs of up to 50GB in size and obtain a citable DOI. Your uploads can be grouped under one or more communities.

Zenodo only

When your research output is not evolving so that there is no point to manage it via Git, you can just upload the to Zenodo only, specify the metadata, and be done. Be sure to comply with the metadata standards and pick the right community or communities to publish under. To do so:

  1. Log in to Zenodo

  2. Click on the New Upload.

  3. Fill in the metadata details in the form that appears.

  4. Upload your files.

  5. Click on Publish.

Manual Zenodo upload of Git release

When your research output is evolving and under Git management, you can make your release citable with a Zenodo DOI, and make it findable and downloadable via Zenodo. To do so manually:

  1. First reserve a DOI by logging in to Zenodo, clicking on the New Upload button, and under the Digital Object Identifier section clicking Reserve DOI.

  2. Paste the DOI into a README or other documentation in your Git-managed file tree. You can turn the DOI into a clickable URL by prefixing it with https://doi.org/. Typically, you will have a section in the documentation called “How to cite” where you want the DOI to appear.

  3. Make a release of the repository as per the instructions in the Tagging or GitHub/GitLab Release Management sections above.

  4. Obtain a ZIP or other archive of the release. When you use GitHub or GitLab release management, such an archive is automatically created and downloadable from the release page. When you tag releases locally, archive a clean worktree corresponding to the release commit, and make sure that the .git holding the local repository data store is excluded.

  5. Proceed as detailed in the prior Zenodo only section, and upload the archive in step 4.

Automatic Zenodo upload of GitHub release

When releasing on GitHub, you can use the Zenodo-GitHub integration to automatically archive your GitHub releases on Zenodo and create a new Zenodo DOI for each release.

Updating Zenodo releases

Zenodo allows you to issue a new versions of your research output as updates under the same record. This is called DOI versioning. Zenodo generates a DOI for the entire series of versions, and also specific DOIs for each version. See the DOI versioning FAQ for details.
