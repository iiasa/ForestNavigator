Configuration

Container technologies provide command line commands to initiate and parameterize the building, running, listing, inspection and so on of images and containers. Using the command line is handy for interactive trialing and testing, but the large number of parameters makes that command lines can get very long. Instead of consolidating these very long command invocations in a shell script or batch file, it is possible to configure containerization through specialized configuration files.

One example thereof we already discussed: the build script used to configure the building of images.

Other aspects of containerization can also be specified in the form of configuration files. These are typically YAML files, a simple declarative repository-friendly text format. Various kinds of configuration items are encoded therein:

  • Build arguments: These are made available to the build script when an image is built. An example use case is to easily create differently configured images via a single build script.

  • Run arguments: These configure how a container is to be run and what external resources such as file volumes or network access it is to be provided with.

  • Build-time environment variables: Environment variables can be picked up by the build script to customize the image being built. These can be specified as name/value pairs or by referencing a file containing a list name/value pair environment variable definitions.

  • Run-time environment variables: Environment variables that are passed into containers to configure the software running therein.

As an example, Docker containerization can be configured through Docker compose files.

Note

Docker compose files configure, amongst other things, “services”. In essence, the term service is simply a synonym for container. Though confusing, it helps to think of it as follows: a running container often provides some kind of service so that it is reasonable to refer to it as a service in a context where the focus is on how containers run. There is though a slight difference: service names are not the same as container names. Unlike container names, they need not be unique. This allows containers sharing a virtual network to contact each other via simple service names descriptive of the service provided instead of complex container names that might contain IDs or instance counts to ensure global uniqueness. This will work without name clashes as long as the service names of containers sharing a network are unique.

Caution

To add to the confusion, Kubernetes too has Service objects. These determine how containers are exposed to the cluster or outside world.

Such configuration files can also be managed with Git. It is often convenient to store the configuration files in the same repository that holds the code of the software being containerized. If the configuration pertains to multiple containers, an umbrella Git repository can be used that includes specific tags of multiple code repositories via Git submodules.

Of course, the very same holds for the configuration files of your model or app: these can also be maintained under Git version control for manageability and reproducibility.

Danger

When the version control repository is public or shared with others, avoid committing configuration settings that are machine-specific or divulge secrets. Replace these with innocuous default values, or manage the secrets separately.
