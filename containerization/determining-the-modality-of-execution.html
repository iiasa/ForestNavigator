Determining the modality of executionÔÉÅ

Before starting with containerization, it is important to clearly determine the modality of execution of the software that you intend to containerize. You can do so by asking yourself and answering questions such as:

  • What dependencies and environment does the software require during its execution?

  • When should the software start?

  • When should the software stop?

  • What services does the software need to access during its execution? For each service consider the following questions:

    • Can the service be run as a separate container deployed on the same ForestNavigator Kubernetes cluster? (preferred if possible because it allows much benefit to be derived from the orchestration functionality provided by Kubernetes).

    • Is the service available only remotely via the Internet?

    • If neither of the above, is there really no other option than deploying the service in the same container? (a last resort).

  • What service does the software provide during its execution?

  • Should the software execute as a job?

    • Taking in inputs at the start and producing outputs at the end.

    • For example, a model scenario run.

  • Does the software need to run continuously?

    • For example, because it provides an on-demand service.

  • What inputs from outside the container does the software process?

  • What output should the software deliver outside of the container?

  • Can or do parts of the software execute in parallel?

    • If so, can these be put in separate containers?

  • Can multiple copies of the software execute in parallel?

    • Can these be run as multiple container instances?

The answers to these questions will obviously constrain how the container images have to be built, what interfaces should be defined, and how the running of the containers is to be orchestrated.
