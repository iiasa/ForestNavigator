Networking

Destinations

Containers can interact to the outside world through what looks like a standard network connection from inside the container. Various destinations can be reached.

Nowhere

Networking can be explicitly disabled for containers. This may be desirable from a security perspective when using images of questionable origin.

The Internet

Typically, containers are provided with network access to the Internet. This allows them to reach cloud services or utilize web APIs. The host computer on which the container engine runs will itself have to have Internet access for this to work.

The host

Running on top of the host operating system, there may be services exposed via a network socket. For example a database. To access host-side services, containers need to utilize a special host address. In case of Docker, this is host.docker.internal. Depending on the service, a semicolon followed by a particular port number will need to be appended to the host address.

Other containers

Containers are isolated from each other by default. To allow them to interact via network connections, some kind of container network can be provided by the container engine. This of course not a physical network with cabling and connectors but rather a software-defined network.

When started, a container can be given access to a particular container network. Multiple container networks can be created so that groups of related containers can each have their own isolated network. Typically, networks are named, and containers can access each other by using their instance name as host name.

Under the hood

The above works transparently from a container perspective because the container engine intercepts and redirects outgoing network traffic.
